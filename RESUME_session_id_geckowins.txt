RESUME — wiring session_id onto GeckoGameWin + building session detail view
============================================================================

Context recap
-------------
We added `session_id` (UUID) as the connective thread between PastMeet,
UserFriendCurrentLiveSesh, and UserFriendLiveSeshLog. Generated at
accept-invite time. Already wired:

  - friends.PastMeet.session_id              (added)
  - users.UserFriendCurrentLiveSesh.session_id  (added, indexed via OneToOne)
  - users.UserFriendLiveSeshLog.session_id   (added, db_index=True, copied
                                             onto log from save() side effect)
  - accept_live_sesh_invite generates UUID, stamps both live-sesh rows,
    creates PastMeet(s), and fires:
      * live_sesh_invite_accepted (host only) — payload includes session_id,
        friend_id, hello_id
      * helloes_updated (host and/or guest if their PastMeet was created)
  - Notification consumer has helloes_updated handler.
  - FE useNotificationsSocket has been updated in
    useNotificationsSocket.updated.ts (root of repo). Copy/paste over the
    existing hook. Delete the staging file after.
  - Backend endpoint added: GET /friends/<friend_id>/live-sessions/
    (FriendLiveSessionsAll). Paginated via MediumPagination + ?nopaginate=true
    escape hatch. Aggregates points from UserFriendLiveSeshPoints (not the
    dead host_points/guest_points columns on the log).
